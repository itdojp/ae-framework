# LLM First-Pass Review Checklist

> 🌍 Language / 言語: English | 日本語

---

## English

Purpose: Provide a consistent, human-approved checklist for the first-pass review of AI-assisted changes before merge.

Scope:
- Use this when an LLM or agent produced code, tests, or documentation.
- This does not replace human review; it standardizes it.

## Required inputs
- Spec or Blueprint (if present): `docs/templates/spec-kit/` or `docs/templates/blueprint/`
- Plan -> Spec template (if used): `docs/templates/plan-to-spec-normalization-template.md`
- DbC definition source (if used): Plan->Spec, Spec/Blueprint, or Context Bundle
- Context Bundle (if used): `docs/guides/context-bundle.md`
- Context Vacuum checklist: `docs/guides/context-vacuum-checklist.md`
- Verification summary: `docs/quality/pr-summary-template.md`
- Gates guide: `docs/quality/verification-gates.md`

## Checklist

### 1) Intent and scope
- The PR summary matches the actual diff scope, with no hidden changes.
- The change aligns with the Spec or Blueprint, or a short rationale explains deviations.
- Any new behavior has explicit acceptance criteria.
- New or changed behavior has explicit preconditions, postconditions, and invariants (DbC) in Plan->Spec or the Context Bundle.

### 2) Evidence and verification
- Required CI checks are green, with Verify Lite as the minimum baseline.
- Optional gates are enabled when risk is high, such as mutation, contract, MBT, or perf/a11y/lh.
- Report artifacts are linked or summarized in the PR.
- Preconditions and postconditions are covered by unit, integration, or negative tests.
- Invariants are covered by at least one independent mechanism, such as property tests, runtime conformance, DB constraints, or equivalent checks.

### 3) Risk and safety
- No security-sensitive change is merged without explicit review notes.
- External API changes have contract or compatibility evidence.
- Performance regressions are ruled out or explicitly accepted.

### 4) Machine-verifying-machine guard
- Tests were not generated by the same model without human review.
- For critical logic, at least one independent check exists, such as manual reasoning, a separate tool, or a targeted test.
- DbC definitions and DbC verification are not both accepted from a single generated artifact without independent confirmation.

### 5) Rollback and ownership
- The rollback path is documented for behavior changes.
- Ownership is clear, including who to contact if the change breaks.

## Decision record (copy into PR comment)

```text
LLM First-Pass Review
- Intent/Scope: [OK/Needs work]
- Evidence/Gates: [OK/Needs work]
- Risk/Safety: [OK/Needs work]
- Machine-verifying-machine guard: [OK/Needs work]
- Rollback/Ownership: [OK/Needs work]
- Notes:
```

## 日本語

目的: AI 支援で生成された変更に対して、merge 前の一次レビューを人間主導で一貫して実施するためのチェックリストを提供します。

対象:
- LLM やエージェントがコード、テスト、ドキュメントを生成した変更に適用します。
- 人間のレビューを置き換えるものではなく、一次レビューの観点を標準化するための文書です。

## 必要な入力
- Spec または Blueprint（存在する場合）: `docs/templates/spec-kit/` または `docs/templates/blueprint/`
- Plan -> Spec template（使用した場合）: `docs/templates/plan-to-spec-normalization-template.md`
- DbC の定義ソース（使用した場合）: Plan->Spec、Spec/Blueprint、または Context Bundle
- Context Bundle（使用した場合）: `docs/guides/context-bundle.md`
- Context Vacuum checklist: `docs/guides/context-vacuum-checklist.md`
- Verification summary: `docs/quality/pr-summary-template.md`
- Gates guide: `docs/quality/verification-gates.md`

## チェックリスト

### 1) 意図とスコープ
- PR summary が実際の diff scope と一致し、隠れた変更がない。
- 変更内容が Spec または Blueprint と整合し、逸脱がある場合は短い理由が説明されている。
- 新しい振る舞いには明示的な acceptance criteria がある。
- 新規または変更された振る舞いに対して、Plan->Spec または Context Bundle に前提条件、事後条件、不変条件（DbC）が明示されている。

### 2) 証跡と検証
- 必須 CI check が green であり、最低限 Verify Lite が通過している。
- mutation、contract、MBT、perf/a11y/lh など、リスクが高い場合に optional gate が有効化されている。
- レポート成果物が PR にリンクされるか、要約されている。
- 前提条件と事後条件が unit、integration、negative test のいずれかで検証されている。
- 不変条件が property test、runtime conformance、DB constraint など、少なくとも 1 つの独立した仕組みで検証されている。

### 3) リスクと安全性
- security-sensitive な変更に対して、明示的な review note なしで merge しない。
- 外部 API 変更には contract または compatibility の証跡がある。
- performance regression は否定されているか、明示的に受容されている。

### 4) 機械が生成したものを機械だけで検証しないためのガード
- テストが同一モデルで生成され、人間のレビューなしに受理されていない。
- critical logic には manual reasoning、別ツール、targeted test など、少なくとも 1 つの独立した検証がある。
- DbC の定義と DbC の検証結果を、独立確認なしに同一の生成成果物だけで受理していない。

### 5) ロールバックと責任分界
- 振る舞い変更に対する rollback path が文書化されている。
- ownership が明確であり、障害時の連絡先が判断できる。
